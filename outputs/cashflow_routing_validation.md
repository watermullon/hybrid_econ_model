# Cashflow Routing Validation

## Routing Logic

The model now routes two generated cashflow sources explicitly:

- real estate net cashflow;
- harvested hedge fund gains.

Each source is split into:

- immediate LP cash distributions;
- hedge fund reinvestment;
- reserve allocation.

The configured percentages must sum to 100% for each source. Validation is handled by Pydantic before a scenario can run.

## Default Assumptions

The default `inputs/model_config.yaml` routing is:

```yaml
cashflow_routing:
  re_cashflow:
    lp_distribution_pct: 0.50
    hf_reinvestment_pct: 0.30
    reserve_pct: 0.20
  hf_harvest:
    lp_distribution_pct: 0.70
    hf_reinvestment_pct: 0.20
    reserve_pct: 0.10
```

These settings reflect the clarified intent that LP hurdle achievement is based on actual cash distributions received, while still allowing part of generated cash to compound in the hedge fund sleeve or build reserves.

## Output Visibility

Annual cashflow output now includes:

- cash generated by source;
- cash routed to LP;
- cash routed to HF;
- cash routed to reserve;
- cumulative reinvestment into HF;
- cumulative cash distributed to LP;
- cumulative cash added to reserve.

Summary output now includes:

- total generated RE cashflow;
- total generated HF harvest;
- total reinvested into HF;
- total distributed to LP;
- total added to reserve;
- cash routing percentages by actual totals;
- years until LP reaches 1x and 2x cash return;
- LP cashflow profile type.

## Behavioural Implications

Routing RE cashflow or HF harvest to LP distributions accelerates LP cash MOIC and can shorten time to the 2x hurdle.

Routing cash to HF reinvestment can improve future HF NAV and compounding, but it does not help the LP cash hurdle until that value is later harvested, distributed, or used for redemption.

Routing cash to reserve improves liquidity and support capacity, but it also delays actual LP cash distributions unless reserve is later used for LP payments.

The Streamlit dashboard now includes routing sliders. Those sliders rerun the existing eight scenarios in memory only and do not save YAML or overwrite model outputs.
